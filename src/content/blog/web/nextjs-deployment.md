Next.js를 배포하면 반복해서 마주치는 불만 세 가지가 있어요.

- 배포는 잘 됐는데 한동안 안 쓰던 페이지의 첫 요청만 유독 느려요.
- Vercel 청구서가 예고 없이 튀어요.
- Cloudflare에 올리려고 보니 OpenNext라는 걸 끼워야 한다고 해요.

세 가지가 따로 노는 문제처럼 보이지만, 원인은 두 개로 정리돼요. 하나는 **런타임 모델**, 다른 하나는 **Next.js의 이식성**입니다. 이 글은 그 둘을 분리해서 설명하고, 그 위에서 "내 상황엔 어디에 올리는 게 맞는지"를 정리합니다.

---

## 1. Next.js가 가끔 느린 진짜 이유

### 콜드스타트는 프레임워크 탓이 아니에요

콜드스타트는 서버리스 런타임의 성질이에요. Vercel이나 AWS Lambda 같은 환경은 요청이 들어올 때마다 함수를 띄워요. 한동안 호출이 없던 함수는 인스턴스가 내려가 있다가, 다음 요청에 다시 부팅됩니다. 그 부팅 시간이 첫 요청에 그대로 더해져요.

Vercel Functions 기준으로 공식 수치는 없지만, 경험상 작은 Node 함수는 수백ms, 무거운 SSR은 수초까지 걸려요. "가끔 느린" 정체가 대부분 이거예요. 코드가 느린 게 아니라, 안 쓰던 함수가 깨어나는 시간이에요.

### OpenNext는 "느림"이 아니라 별개의 문제예요

OpenNext를 콜드스타트와 같은 묶음으로 이해하면 헷갈려요. 이건 이식성 문제예요.

Next.js는 Vercel에서 가장 매끄럽게 동작하도록 최적화돼 있어요. SSR, ISR, 이미지 최적화, 미들웨어가 Vercel 위에서는 자연스럽게 동작하지만, Cloudflare Workers나 Lambda처럼 다른 서버리스/엣지 환경에 올리려면 Next.js의 출력을 그 런타임에 맞게 재구성해야 해요. OpenNext가 그 어댑터예요. Cloudflare에 Next.js를 올릴 때 OpenNext를 거치는 이유가 이거예요.

정리하면 두 통증의 원인은 달라요.

- **콜드스타트** → 서버리스라는 `런타임` 때문
- **OpenNext** → Next.js를 Vercel 밖으로 `이식`하기 때문

이 구분이 이 글의 나머지를 관통합니다.

## 2. 배포 환경을 가르는 한 축: 런타임 모델

배포 옵션은 많아 보이지만, 실제 결정은 한 축 위에서 갈려요. **scale-to-zero냐, always-on이냐**입니다.

**scale-to-zero (서버리스 / 워커)**
트래픽이 없으면 인스턴스를 0으로 내려요. 첫 요청에 깨어나므로 콜드스타트가 있어요. 대신 유휴 시간에는 과금되지 않습니다. Vercel, Netlify, Cloudflare Workers가 여기 속해요.

**always-on (컨테이너 / 서버)**
프로세스가 계속 떠 있어요. 콜드스타트가 없어요(배포 직후 재시작 시간은 있지만 콜드스타트보다 짧아요). 대신 트래픽이 없어도 인스턴스가 떠 있는 만큼 과금됩니다. Railway, Render, Fly.io, 그리고 직접 운영하는 VPS가 여기 속해요.

이 한 줄이 콜드스타트 유무, 비용 구조, 확장 방식을 동시에 결정합니다.

### 핵심: always-on Node 서버면 두 문제가 같이 사라져요

여기서 1장의 두 통증이 한 번에 해결되는 지점이 나와요.

Next.js를 `output: 'standalone'`으로 빌드하면 단독 실행 가능한 최소 Node 서버가 나와요. 이걸 계속 떠 있는 환경(컨테이너, VM, VPS)에 올리면:

- 프로세스가 살아 있으니 **콜드스타트가 없어요.**
- 일반 Node 서버에서 도는 것이므로 **OpenNext 같은 어댑터가 필요 없어요.** App Router, Server Actions, ISR, 이미지 최적화가 전부 네이티브로 동작해요.

OpenNext는 Next.js를 서버리스/엣지에 _욱여넣을 때만_ 필요해요. always-on으로 가면 처음부터 등장할 일이 없습니다.

대신 트레이드오프가 있어요. always-on은 유휴 시간에도 비용이 나가고, CDN·무중단 배포·SSL 같은 걸 직접 챙겨야 해요. 그 부담을 누가 지느냐가 다음 두 선택지를 가릅니다.

## 3. 선택지 ①: 자체 호스팅 (VPS · 클라우드 VM + Coolify)

자체 호스팅이라고 하면 VPS만 떠올리기 쉽지만, 핵심은 "SSH로 접속해 내가 통제하는 리눅스 서버 한 대"예요. Hetzner 같은 VPS든, AWS EC2·GCP Compute Engine·Azure VM 같은 클라우드 VM이든 전부 여기 해당해요. 공통점은 통제권이 가장 크고 비용이 가장 낮은 대신, Vercel이 공짜로 해주던 걸 직접 떠안는다는 거예요. CDN, ISR 캐시 공유, 무중단 배포, 오토스케일, SSL 갱신, 로그, 모니터링. 순수 EC2에 손으로 올리면 이 운영 부담이 종속성보다 무거워질 수 있어요.

이 부담을 덜어주는 게 Coolify예요. 오픈소스 셀프 호스팅 PaaS로, 내 서버 위에 Vercel 같은 경험을 올려줘요. Git 연동 배포, PR 프리뷰 URL, 자동 SSL, 모니터링, 원클릭 DB를 제공하고, 벤더 종속이 없어요. EC2든 Hetzner든 SSH 되는 리눅스 서버면 동일하게 돌아갑니다.

요구사항은 최소 2코어 / 2GB RAM / 30GB 디스크예요. 단 Coolify 자체가 아무것도 배포 안 한 상태에서 ~1GB를 먹으므로, 빌드 중 OOM을 피하려면 4GB를 권장해요.

**비용 선택지**

- **Oracle Cloud Always Free**

  ARM 4 OCPU / 24GB RAM / 200GB / 월 10TB 트래픽, 평생 무료. 학습·개인용으론 이게 가장 관대해요. 단점은 설정 러닝커브가 가파르고, ARM 인스턴스 용량이 인기 리전에서 잘 안 잡힐 때가 있다는 거예요.

- **Hetzner CX22**

  2vCPU / 4GB, 월 €5 정도. 무료 티어의 용량 불안정성이 싫다면 가성비가 가장 좋아요.

EC2 같은 클라우드 VM도 Coolify 입장에선 똑같은 리눅스 서버라 그대로 동작해요. 다만 비용 구조가 달라요. 같은 4GB 기준 EC2(t4g.medium)는 월 \$24 안팎으로 Hetzner보다 비싸고, 아웃바운드 트래픽에 GB당 약 \$0.09가 따로 붙어요. 그래서 순수하게 "싸게"가 목적이면 Oracle·Hetzner가 유리하고, EC2는 이미 AWS를 쓰고 있거나 RDS·S3 같은 AWS 서비스와 붙여야 할 때 합리적이에요. 인스턴스는 ARM(t4g) 계열을 고르면 비용이 줄고, Coolify가 ARM64를 지원해요. 보안 그룹에서 22·80·443·8000 포트를 열고, IP가 바뀌지 않게 Elastic IP를 붙여두면 됩니다.

점진적 확장도 자연스러워요. 정적 사이트를 무료로 시작 → SSR이 필요해지면 저가 VPS + Coolify → 트래픽이 늘면 인스턴스 증설. 산출물이 Docker 컨테이너라 나중에 서버만 갈아끼우면 돼요.

## 4. 선택지 ②: 관리형 PaaS

운영에 시간을 쓰기 싫고 바로 배포가 목적이면 관리형이에요. 같은 관리형이라도 1장의 런타임 축에서 갈려요.

- **Vercel**

  Next.js 전용기. git push 한 번에 빌드·배포, PR마다 프리뷰, 엣지 배포·이미지 최적화 번들. DX는 이 중 최강이에요. 단 scale-to-zero라 콜드스타트가 있고, Pro 기본 대역폭 1TB를 넘으면 GB당 \$0.15 안팎(리전별 차등)이 붙어요(무료 Hobby 티어는 100GB). 트래픽 스파이크에 비용이 비선형으로 튀어요. \$20 예산이 봇 트래픽 한 번에 수백 달러가 되는 사례가 흔해요.

- **Netlify**

  프론트 중심으로 Vercel과 성격이 비슷하고, 과금이 크레딧 기반으로 바뀌었어요. 정적·Jamstack에 강해요.

- **Railway**

  always-on이라 콜드스타트가 없어요. 시각적 프로젝트 캔버스에 원클릭 DB(Postgres·Redis 등), 사용량 기반 과금. DX가 좋아 빠른 프로토타입에 강해요. 단일 리전 중심이고, 영구 무료 서버는 없어요(가입 시 $5 체험 크레딧만 제공돼요).

- **Render**

  이 비교군에서 유일하게 신용카드 없이 쓰는 영구 무료 티어가 있어요. 단 무료 웹 서비스는 15분 비활성 후 스핀다운되어 깨어나는 데 약 30~60초가 걸려요(정적 사이트는 sleep 없음). 실서비스는 $7/월 always-on부터, Postgres 별도 $7/월이에요. 고정 요금이라 예산 예측이 쉽고 멀티 리전을 지원해요.

- **Fly.io**

  35개 이상 리전에 컨테이너를 사용자 근처에 띄워요. 원시 컴퓨팅 단가가 가장 싸고(1vCPU/2GB 약 $10.70/월), scale-to-zero(auto_stop) 옵션과 GPU·Docker 제어를 제공해요. auto_stop을 켜면 콜드스타트가 생겨요. 제어가 큰 만큼 러닝커브도 가팔라요.

|            | Vercel              | Railway      | Render         | Fly.io                |
| ---------- | ------------------- | ------------ | -------------- | --------------------- |
| 런타임     | scale-to-zero       | always-on    | always-on      | 엣지 (auto-stop 가능) |
| 콜드스타트 | 있음                | 없음         | 없음(유료)     | auto_stop 시 있음     |
| 영구 무료  | 정적/소규모         | ❌ ($5 체험) | ✅ (CC 불필요) | ❌ (체험만)           |
| 과금       | Active CPU + 대역폭 | 사용량       | 고정 요금      | 사용량                |
| 리전       | 글로벌 CDN          | 단일         | 멀티           | 멀티(35+)             |
| 강점       | Next.js DX          | 편의성       | 예측 가능성    | 글로벌 지연·비용      |

전형적인 소규모 풀스택(웹 1 + 워커 1 + Postgres 1, 보통 트래픽) 월 비용은 Render·Railway가 $7\~15, Fly.io가 $10\~20, Vercel이 사용 방식에 따라 $20\~200 수준이에요.

## 5. 상황별 추천

결정은 세 축의 곱이에요. **트래픽 규모 × 운영 여력 × 비용 예측성.**

- **학습 · 개인 프로젝트, 운영도 배우고 싶다** → Oracle Always Free 또는 Hetzner + Coolify. 비용이 0~월 몇 천 원이고 통제권이 가장 커요.
- **빠른 시작과 DX가 우선** → Railway. 콜드스타트 없이 원클릭 DB까지 가장 빠르게 production에 올려요.
- **비용을 예측 가능하게 묶고 싶다** → Render. 고정 요금이라 청구서 쇼크가 없어요.
- **사용자가 여러 지역에 있고 지연이 중요** → Fly.io. 멀티 리전이 존재 이유예요.
- **Next.js 프론트 + 간단한 백엔드, 트래픽이 작고 예측 가능** → Vercel. 이 경우엔 DX 프리미엄이 값을 해요.

흔한 실전 구성은 프론트는 Vercel/Netlify 무료 티어, 백엔드 API+DB는 Railway나 Render로 분리하는 방식이에요.

## 6. Next.js만의 문제일까? — 다른 프레임워크

1장에서 본 OpenNext 문제는 Next.js 고유예요. 다른 풀스택 프레임워크는 이식성을 1st-party로 해결해요.

- **Nuxt**

  내부에 Nitro라는 서버 엔진이 있고, 15개 이상의 프리셋으로 빌드 출력을 각 호스팅 타깃에 맞게 변환해요. 타깃을 바꿔도 코드를 고칠 필요가 없고, CI/CD에서 제공자 환경을 자동 감지해 프리셋을 잡아줘요. 기본 node-server 프리셋이 바로 실행 가능한 Node 서버를 만들어요. OpenNext 같은 외부 의존이 없어요.

- **SvelteKit**

  공식 어댑터로 해결해요. adapter-node(단독 Node 서버), adapter-static, adapter-vercel, adapter-cloudflare, adapter-auto가 전부 공식 제공이고 Svelte 코어 팀이 직접 관리해요. 번들이 가볍고 메모리 사용이 적어 가장 싼 VPS에서도 잘 돌아가요.

단, 콜드스타트는 프레임워크와 무관하다는 점은 그대로예요. Nuxt든 SvelteKit이든 서버리스에 올리면 콜드스타트가 생기고, always-on에 올리면 없어져요. 해결책은 동일해요 — long-running 서버 타깃을 고르면 됩니다.

- **정적 사이트(SSG)**

  서버 렌더링이 필요 없다면 가장 단순하고 싸요. 프리렌더해서 Cloudflare Pages나 GitHub Pages에 올리면 완전 무료에 콜드스타트도 없어요. 세 프레임워크 모두 라우트별로 정적/SSR을 섞는 하이브리드 렌더링을 지원하므로, 서버가 꼭 필요한 라우트만 SSR로 두면 서버 비용 자체를 최소화할 수 있어요.

정리하면, **이식성 문제는 Next.js 고유지만 배포 원리(런타임 축)는 모든 프레임워크에 공통**이에요.

## TL;DR

- "가끔 느림"은 콜드스타트이고, 이건 서버리스 `런타임`의 성질이에요. OpenNext는 별개로, Next.js를 Vercel 밖으로 `이식`할 때 내는 세금이에요.
- `output: 'standalone'`으로 빌드해 always-on Node 서버에 올리면 콜드스타트와 OpenNext 문제가 둘 다 사라져요. 대신 운영을 직접 떠안아요.
- 운영을 배우거나 비용을 0에 가깝게 하고 싶으면 `Oracle/Hetzner + Coolify`. 운영에 시간 쓰기 싫으면 `Render(예측 가능)`나 `Railway(편의성)`, 글로벌이면 `Fly.io`, Next.js 프론트면 `Vercel`.
- 전부 Docker 기반이라 갈아타기 쉬워요. 완벽한 선택을 고민하느라 시간 쓰지 말고, 하나 골라 배포한 다음 실제 트래픽이 드러나면 옮기세요.
- Nuxt·SvelteKit은 1st-party 어댑터로 이식성을 해결하므로 OpenNext 같은 고민이 없어요. 정적 사이트면 Cloudflare Pages·GitHub Pages로 무료예요.
